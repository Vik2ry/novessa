# Render Deployment Guide

This guide will help you deploy the Novessa application to Render.

## Prerequisites

- Render account (https://render.com)
- GitHub repository connected to Render
- Environment variables ready

## Backend (Django API) - `apps/api`

### Automatic Deployment

The project includes a `render.yaml` configuration file that automates the deployment process.

**To deploy:**

1. Push your code to GitHub
2. In Render dashboard, create a new service from your GitHub repository
3. Select the `render.yaml` configuration when prompted
4. Render will automatically use the configuration from this file

### Environment Variables to Set

After Render creates the services, configure these environment variables in the Render dashboard under your `novessa-api` service settings:

#### Email Configuration (Required for sending emails)
- `EMAIL_HOST_USER`: Your Gmail address or email provider username
- `EMAIL_HOST_PASSWORD`: Your email provider password or app-specific password

#### Payment Processing (Required for donations)
- `PAYSTACK_PUBLIC_KEY`: Your Paystack public key
- `PAYSTACK_SECRET_KEY`: Your Paystack secret key

#### Email Service Integration (Optional, for advanced email features)
- `MAILGUN_API_KEY`: Your Mailgun API key
- `MAILGUN_DOMAIN`: Your Mailgun domain
- `MAILGUN_WEBHOOK_SIGNING_KEY`: Your Mailgun webhook signing key

#### Auto-Generated
- `SECRET_KEY`: This will be automatically generated by Render
- `DATABASE_URL`: This will be automatically created and linked from PostgreSQL service

### Database

The configuration includes:
- **PostgreSQL Database**: `novessa-postgres` (free tier)
  - Automatically created and linked to the API service
  - Connection string stored in `DATABASE_URL` environment variable

**Database Migrations**: Automatically run during deployment via the `preDeployCommand` and `buildCommand`

### Health Check

The API includes a health check endpoint at `/api/v1/health/` which Render uses to verify the service is running properly.

### Static Files

Static files are:
- Collected during build via `python manage.py collectstatic --noinput`
- Served through WhiteNoise (no separate static file service needed)
- Compressed for better performance

## Frontend (Next.js) - `apps/web`

The frontend can be deployed separately to Render or Vercel. Update the `NEXT_PUBLIC_API_BASE_URL` environment variable to point to your Render API:

```
NEXT_PUBLIC_API_BASE_URL=https://novessa-api.onrender.com/api/v1
```

## Troubleshooting

### Build fails
- Check that all dependencies in `requirements.txt` are compatible
- Verify Python version in `.python-version` matches runtime requirements
- Review build logs in Render dashboard

### Database connection errors
- Ensure `DATABASE_URL` is properly set and linked
- Check PostgreSQL service is running
- Verify database migrations completed successfully

### Static files not loading
- Check that `STATIC_URL` is correctly configured (default: `/static/`)
- Verify `STATIC_ROOT` path is correct
- Ensure `collectstatic` command runs successfully during build

### Email not sending
- Verify email credentials are correct
- Check spam folder for test emails
- Review error logs in Render dashboard
- For Gmail: Use an [App Password](https://myaccount.google.com/apppasswords), not your regular password

## Monitoring

Access your API:
- Production: `https://novessa-api.onrender.com`
- API endpoint: `https://novessa-api.onrender.com/api/v1`
- Health check: `https://novessa-api.onrender.com/api/v1/health/`
- Admin panel: `https://novessa-api.onrender.com/admin/`

View logs in the Render dashboard under your service's "Logs" tab.

## Redeploying

**To redeploy after code changes:**

1. Push changes to the GitHub branch connected to Render
2. Render automatically redeploys (if auto-deploy is enabled)
3. Monitor the deployment in Render dashboard

**To trigger a manual redeploy:**

1. Go to your service in Render dashboard
2. Click "Manual Deploy" → "Deploy latest commit"

## Local Development vs Production

The application automatically switches between environments using the `DJANGO_SETTINGS_MODULE` environment variable:

- **Local**: Uses `config.settings.development` (SQLite, DEBUG=True)
- **Production (Render)**: Uses `config.settings.production` (PostgreSQL, DEBUG=False, HTTPS)

Local development is unaffected by Render configuration.
